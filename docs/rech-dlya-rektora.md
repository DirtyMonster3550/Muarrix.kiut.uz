**Формат:** 5–7 минут устно + при необходимости слайды по разделам ниже

## 1. Вступление (30 секунд)

Уважаемый ректор,

мы представляем **единую веб-платформу** для журнала **«Центрально-азиатский журнал STEM»** (ISSN 2181-2934). Это официальный сайт издания: информация для авторов и читателей, **онлайн-подача рукописей**, **экспертиза**, **публикация** и **открытый архив** выпусков — всё в одном месте, на трёх языках: **узбекский, русский и английский**.

## 3. На каком языке написан сайт (технологии)

Кратко для нетехнической аудитории:

> **Сайт написан на JavaScript (Node.js)** — это распространённый язык для веб-серверов и современных приложений.  
> **Страницы** — HTML, CSS и JavaScript в браузере (без тяжёлого отдельного «мобильного приложения»).  
> **Серверная часть** — фреймворк **Express** (Node.js).  
> **База данных** — **SQLite** (файл `db/stem.db`): пользователи, статьи, выпуски, уведомления, журнал действий админа.

**Структура проекта:**

```
STEM/
├── server.js          — запуск сервера
├── routes/            — API: авторизация, статьи, эксперты, админ
├── db/                — база SQLite + инициализация таблиц
├── lib/               — архив, публикация, обложки выпусков
├── public/            — сайт: страницы, стили, переводы (UZ/RU/EN)
├── uploads/           — загруженные рукописи (доступ по роли)
└── public/archives/   — опубликованные материалы для гостей
```

**Запуск на компьютере или сервере:** установить Node.js → `npm install` → `npm start` → сайт на порту 3000 (или за reverse-proxy на 80/443).

## 4. Как устроена работа (бизнес-процесс)

1. **Автор** регистрируется, загружает статью (DOC/DOCX), выбирает выпуск STEM.  
2. **Технический эксперт** проверяет оформление, назначает **редакционного эксперта**, одобряет или возвращает.  
3. **Редакционный эксперт** (назначенный) даёт решение по содержанию.  
4. **Администратор** публикует одобренную статью в выпуске — файл попадает в **публичный архив**.  
5. **Любой посетитель** без входа открывает «Архив» и читает/скачивает материалы.

В системе **один администратор**; роли экспертов и авторов разделены.

## 5. Как появляется профиль администратора на сервере

Это важный вопрос для развёртывания.

**При первом запуске** программа создаёт файл базы данных и проверяет: есть ли пользователь с ролью `admin`.

- **Если админа нет** — создаётся **один** аккаунт:
  - e-mail: из переменной окружения `ADMIN_EMAIL` (по умолчанию `admin@kiut.uz`);
  - пароль: из `ADMIN_PASSWORD` (если не задан — временный пароль из кода; **на сервере обязательно задать свой** в файле `.env`).

- **Если админ уже есть** (база перенесена с другого компьютера) — **новый не создаётся**, используются существующие логин и пароль.

**На production рекомендуем:**

1. Скопировать проект на сервер.  
2. Создать файл `.env` по образцу `.env.example`:
   - `ADMIN_EMAIL=…` — рабочая почта ответственного редактора/IT  
   - `ADMIN_PASSWORD=…` — надёжный пароль  
   - `JWT_SECRET=…` — длинная случайная строка (безопасность сессий)  
3. Выполнить `npm install` и `npm start` (или через PM2/systemd).  
4. **Один раз** войти на `/login.html` и при необходимости сменить пароль в настройках / создать экспертов через админку.  
5. **Не публиковать** `.env` и файл базы в открытый доступ; делать резервные копии `db/stem.db`.

**Перенос с текущего компьютера на сервер:** можно скопировать всю папку `db/` вместе с базой — тогда все пользователи, статьи и выпуски сохранятся; отдельно «создавать админа» не нужно.

## 6. Безопасность (кратко)

- Пароли хранятся **в зашифрованном виде** (bcrypt).  
- Файлы рукописей в `/uploads` — **не публичные**; доступ у автора, экспертов и админа.  
- Опубликованные материалы — только в `/archives/…` для всех.  
- Ограничение частоты запросов (защита от перебора пароля).  
- Журнал действий администратора (audit log).

## 7. Что показать живьём (демо 2–3 минуты)

1. **Главная** — журнал, текущий выпуск, кнопка «Читать выпуск».  
2. **Архив** — выпуск → список статей → открытие файла **без входа**.  
3. **Регистрация / кабинет автора** — подача статьи.  
4. **Панель эксперта** — очередь, скачать, одобрить/отклонить (крупный интерфейс для старшего возраста).  
5. **Админка** — выпуски, публикация, синхронизация архива.

## Шпаргалка: ответы на возможные вопросы

| Вопрос | Ответ |
|--------|--------|
| Это WordPress? | Нет, **собственная разработка** на Node.js — под наш процесс рецензирования. |
| Где хранятся данные? | В файле SQLite на сервере; при росте можно перейти на PostgreSQL. |
| Сколько стоит хостинг? | Любой VPS с Node.js; SQLite не требует отдельного сервера БД. |
| PDF принимают? | Сейчас автор загружает **DOC/DOCX**; в архиве могут быть и PDF прошлых выпусков. |
| Кто админ? | Один аккаунт; создаётся при первом запуске или из перенесённой базы. |
